What is Module Federation?

Module Federation allows separate applications to share code at runtime

  • üè† Host Application: Loads and orchestrates remotes
  • üì° Remote Applications: Independent micro-frontends
  • ‚ö° Lazy Loading: Load remotes only when needed
  • üîÑ Runtime Integration: No build-time coupling

Deployment Process

  1. Build All Applications 
    npm run build:prod
  2. Deploy Host Application

    Deploy dist/apps/week/ to https://my-awesome-week/

  3. Deploy Remote Applications
    • Deploy dist/apps/monday/ to https://monday.my-awesome-week/
    • Deploy dist/apps/tuesday/ to https://tuesday.my-awesome-week/
    • And so on...

Server Configuration (Nginx)


# Host application
server {
    listen 443 ssl;
    server_name my-awesome-week;
    
    location / {
        root /var/www/week;
        try_files $uri $uri/ /index.html;
        
        # Enable CORS for module federation
        add_header Access-Control-Allow-Origin *;
        add_header Access-Control-Allow-Methods 'GET, POST, OPTIONS';
        add_header Access-Control-Allow-Headers 'Content-Type';
    }
}

# Monday remote
server {
    listen 443 ssl;
    server_name monday.my-awesome-week;
    
    location / {
        root /var/www/monday;
        try_files $uri $uri/ /index.html;
        add_header Access-Control-Allow-Origin *;
    }
}

How It Works in Production

üöÄ Initial Load

User visits https://my-awesome-week/
‚Üí Only the host application loads
‚Üí No remote applications loaded initially

üì° Route Navigation

User navigates to /monday
‚Üí Angular router triggers: import('monday/Routes')
‚Üí Webpack fetches from https://monday.my-awesome-week/
‚Üí Remote module loads and renders

Performance & Caching

  • üíæ Independent Caching: Each remote cached separately
  • üîÑ Independent Updates: Update remotes without touching host
  • ‚ö° Lazy Loading: Users only download what they visit
  • üéØ Targeted Deployments: Deploy only changed applications
